# 問題 4.32 解答

## 結論

3 章のストリームと 4.2.3 節の遅延リストの違いは、どこまで遅延されるか。
3 章のストリームでは、`cons-stream` は `cdr` だけを遅延する。
よって、

```scheme
(cons-stream <car> <cdr>)
```

の `<car>` はその場で評価され、`<cdr>` だけが必要になるまで評価されない。
一方、4.2.3 節の遅延リストでは、遅延評価器の上で `cons` を通常の手続きとして定義している。

```scheme
(define (cons x y)
  (lambda (m) (m x y)))
```

遅延評価器では合成手続きの引数が遅延されるので、`x` も `y` も必要になるまで評価されない。
つまり、`car` と `cdr` の両方が遅延される。

## 例 1: 使わない `car` の評価を不要にできる

例えば、先頭要素に評価できない式を置く。

```scheme
(define xs
  (cons (/ 1 0)
        (cons 1 '())))

(cdr xs)
```

4.2.3 節の遅延リストでは、`(/ 1 0)` は `xs` を作る時点では評価されない。
また、ここで要求しているのは `cdr` なので、`car` に入っている `(/ 1 0)` はまだ評価されない。
したがって、`car` を使わない限り、このリストの残りである `cdr` を扱うことができる。
これに対して 3 章のストリームでは、

```scheme
(cons-stream (/ 1 0)
             (cons-stream 1 the-empty-stream))
```

のように書くと、`cons-stream` は `cdr` だけを遅延するので、先頭要素の `(/ 1 0)` はストリームのセルを作る時点で評価されてしまう。
したがって、`cdr` だけを使いたい場合でもエラーになる。

## 例 2: 捨てられる要素の計算を避ける

重い計算を各要素に持つリストを考える。

```scheme
(define xs
  (cons (very-expensive-computation 0)
        (cons (very-expensive-computation 1)
              (cons (very-expensive-computation 2)
                    '()))))

(list-ref xs 2)
```

4.2.3 節の遅延リストでは、`list-ref` が `cdr` をたどって第 2 要素に到達しても、それまでの `car` は強制されない。
したがって、本当に必要な第 2 要素のみの計算で済む。

3 章のストリームでは、各セルの `car` はそのセルが構成される時点で評価される。
ストリームの先頭から順に `cdr` をたどる場合でも、たどったセルの `car` の計算は評価される。
必要な第 2 要素だけでなくそれまでの要素の評価も行われてしまう。
