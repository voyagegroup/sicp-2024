# 問題 4.30 解答

## a

`for-each` の定義されている。

```scheme
(define (for-each proc items)
  (if (null? items)
      'done
      (begin (proc (car items))
             (for-each proc (cdr items)))))
```

ここで `begin` の最初の式は

```scheme
(proc (car items))
```

これは単なる変数参照ではなく**手続き適用そのもの**なので、`eval-sequence` がこれを `eval` した時点でその適用は実行される。

今回 `proc` に渡しているのは

```scheme
(lambda (x) (newline) (display x))
```

この手続きの本体では `newline` と `display` が実際に呼ばれる。したがって、副作用は式の値があとで使われるかどうかに関係なく、その式を評価した時点で起きる。

つまり、元の `eval-sequence` は並びの途中の式を「強制」はしないが、「評価」はしている。
`(proc (car items))` の副作用はその評価の中で発生するので、`for-each` は正しく

```text
57
321
88
```

を印字する。

## b

- 元々の `eval-sequence`
  - `(p1 1)` は `(1 2)`
  - `(p2 1)` は `1`
- Cy の変更後の `eval-sequence`
  - `(p1 1)` は `(1 2)`
  - `(p2 1)` は `(1 2)`

### `(p1 1)`

```scheme
(define (p1 x)
  (set! x (cons x '(2)))
  x)
```

`p1` の本体の最初の式

```scheme
(set! x (cons x '(2)))
```

は、`eval` するだけで代入が実行される。したがって `x` は `(1 2)` に更新される。その次の `x` を評価すると `(1 2)` が返る。

よって `(p1 1)` は、元の評価器でも Cy の変更後でも `(1 2)` になる。

### `(p2 1)` を元の `eval-sequence` で評価した場合

```scheme
(define (p2 x)
  (define (p e)
    e
    x)
  (p (set! x (cons x '(2)))))
```

`p2` は `p` を

```scheme
(p (set! x (cons x '(2))))
```

で呼び出す。遅延評価器では、`p` の引数 `e` はすぐには評価されず、`(set! x (cons x '(2)))` を表すサンクとして束縛される。

`p` の本体は

```scheme
e
x
```

である。元の `eval-sequence` は並びの途中の式 `e` を `eval` するだけなので、ここでは **`e` に束縛されたサンクそのものを取り出すだけ** で終わる。`actual-value` ではないので、そのサンクは強制されず、中に入っている

```scheme
(set! x (cons x '(2)))
```

は実行されない。

したがって `x` は最初の値 `1` のままであり、最後の式 `x` の値として `1` が返る。

よって、元の `eval-sequence` では

```scheme
(p2 1) => 1
```

である。

### `(p2 1)` を Cy の変更後の `eval-sequence` で評価した場合

Cy の提案は、並びの途中の式を

```scheme
(actual-value (first-exp exps) env)
```

で扱う、というものだった。

すると `p` の本体の最初の式 `e` は、単にサンクとして取り出されるのではなく、**強制される**。その結果

```scheme
(set! x (cons x '(2)))
```

が実行され、`x` は `(1 2)` に更新される。

その後、最後の式 `x` を評価すると `(1 2)` が返る。

よって、Cy の変更後では

```scheme
(p2 1) => (1 2)
```

になる。

## c

Cy の変更は a の `for-each` の振舞いに影響しない。

`for-each` の並びの途中の式

```scheme
(proc (car items))
```

は、もともと `eval` した時点で手続き適用として実行され、副作用を起こす式である。
この式は「サンクを返して、それを強制しないと副作用が起きない」という種類の式ではない。

したがって、ここを

- `eval` で評価しても
- `actual-value` で強制しても

観測できる振舞いは同じであり、各要素の印字はそのまま行われる。

## d

本文の解決法の方がよいと考える。
遅延評価では、式はその値が必要になるまで評価されない、というのが基本方針。
ところが Cy の提案では、式が並びの途中にあるというだけで強制が起こる。
これは「値が必要だから評価する」のではなく、「その場所に書かれているから評価する」という規則であり、遅延評価の考え方から少し外れている。
