# 問題 4.28 解答

`eval` が `apply` に渡す前に演算子を `eval` ではなく `actual-value` で評価しなければならないのは、**演算子の位置にサンクが現れることがある**からである。

例えば次の式を考える。

```scheme
((lambda (f) (f 3 4))
 (if true + -))
```

遅延評価器では、合成手続き

```scheme
(lambda (f) (f 3 4))
```

に引数を渡すとき、引数

```scheme
(if true + -)
```

はすぐには評価されず、サンクとして束縛される。

したがって手続き本体の

```scheme
(f 3 4)
```

を評価する段階では、`f` に束縛されているのはまだ `+` そのものではなく、`(if true + -)` を表すサンクである。

ここで演算子 `f` を単に `eval` すると、その結果はサンクのままである。そのまま `apply` に渡すと、`apply` はそのオブジェクトを基本手続きとも合成手続きとも認識できず、正しく手続きを適用できない。

一方、演算子を `actual-value` で求めれば、`f` に束縛されているサンクが強制される。すると

```scheme
(if true + -)
```

が評価されて `+` が得られるので、

```scheme
(+ 3 4)
```

として正しく計算できる。

したがって、遅延評価器では演算子の位置にある式も必要に応じて強制しなければならない。そのため `eval` は `apply` に渡す前に演算子を `actual-value` で評価する必要がある。
