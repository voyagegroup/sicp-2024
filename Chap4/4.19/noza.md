# 問題 4.19 解答

対象の式:

```scheme
(let ((a 1))
  (define (f x)
    (define b (+ a x))
    (define a 5)
    (+ a b))
  (f 10))
```

## どの見方を支持するか

私は **Alyssa の見方**（問題 4.16 の実装方針）を支持する。  
理由は、内部定義の「同時有効範囲」を機械的かつ一貫して実装できるからである。

4.16 の掃き出しでは、`f` の本体は概ね次に変換される:

```scheme
(let ((b '*unassigned*)
      (a '*unassigned*))
  (set! b (+ a x))
  (set! a 5)
  (+ a b))
```

ここで `b` を計算する時点では `a` は `*unassigned*` なのでエラーになる。  
したがって結果は **エラー**。

## Ben と Eva の見方について

- Ben（16）: 逐次 `define` をそのまま実行すると `b=11`, `a=5` で 16。  
  ただしこれは「内部定義は本体全体で有効」という規則と相性が悪い。

- Eva（20）: `a` と `b` を同時定義とみなし、`b` の評価に `a=5` を使う。  
  直感としては理解できるが、標準的な掃き出し規則（4.16）とは異なる意味づけが必要。

## Eva の好む挙動は実装できるか

**可能。** ただし 4.16 より複雑な仕組みが必要になる。

1. まず内部定義の変数をすべて束縛する。  
2. 定義右辺の依存関係（どの変数がどれに依存するか）を解析する。  
3. 依存グラフのトポロジカル順で評価・代入する（この例では `a` を先に、次に `b`）。  

この方式ならこの例は `a=5`, `b=15` となり結果は 20 になる。  
ただし一般には、循環依存・副作用・評価順の問題があり、仕様も実装も難しくなる。

## 結論

- 本文と 4.16 の方針に整合する答えとしては **Alyssa（エラー）** を採るのが自然。  
- **Eva 的意味論（20）も設計は可能**だが、別の（より強い）内部定義規則を導入する必要がある。
